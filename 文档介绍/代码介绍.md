# 关键代码介绍

## 1. 文档目标

本文档用于给论文正文或附录提供可直接引用的关键代码片段，并配套给出简明说明。

为了便于论文排版，本文档中的代码遵循两个原则：

- 只保留核心逻辑，省略无关细节
- 优先选择最能体现系统设计思路的片段

其中：

- 硬件部分代码主要来自 `Arduino/esp32.ino`
- 软件部分代码主要来自 `backend/` 与 `frontend/`

部分硬件示例为了适应论文展示，做了适度整理，但字段命名、流程和仓库实现保持一致。

## 2. 论文使用建议

如果论文篇幅有限，建议优先选以下代码：

1. 硬件端采集与 MQTT 上报

```cpp
const int GPS_RX_PIN = 16;
const int GPS_TX_PIN = 17;
```

### 3.2 传感器采集与 GPS 读取代码

来源：

- `Arduino/esp32.ino`

论文展示版代码：

```cpp
void readGpsStream() {
  while (gpsSerial.available() > 0) {
    gps.encode(gpsSerial.read());
  }
}

bool readSensors(float& temperature, float& humidity, float& pressure) {
  sensors_event_t humidityEvent;
  sensors_event_t temperatureEvent;

  if (!sht4.getEvent(&humidityEvent, &temperatureEvent)) {
    return false;
  }

  temperature = temperatureEvent.temperature;
  humidity = humidityEvent.relative_humidity;
  pressure = bmp.readPressure() / 100.0F;
  return true;
}
```

说明：

- `readGpsStream()` 持续从串口读取 NMEA 数据，并交由 `TinyGPS++` 解析。
- `readSensors()` 同时读取温度、湿度和气压数据，其中气压值由 Pa 转换为 hPa。
- 这一步是整套系统的数据源头，后端的异常检测、轨迹展示和哈希计算都依赖这里采集到的原始数据。

论文可写描述：

“设备端通过 I2C 总线读取温湿度与气压传感器数据，同时通过串口异步接收 GPS 定位信息，形成环境监测数据与位置数据的双通道采集结构。”

### 3.3 MQTT 上报代码

来源：

- `Arduino/esp32.ino`

论文展示版代码：

```cpp
String buildPayload(
  const String& deviceId,
  float temperature,
  float humidity,
  float pressure
) {
  DynamicJsonDocument doc(512);
  doc["device_id"] = deviceId;
  doc["uptime"] = millis() / 1000;
  doc["timestamp"] = currentTimestampText();
  doc["ntp_synced"] = true;

  JsonObject sensors = doc.createNestedObject("sensors");
  sensors["temperature"] = String(temperature, 2);
  sensors["humidity"] = String(humidity, 2);
  sensors["pressure"] = String(pressure, 2);
  sensors["valid"] = true;

  JsonObject gpsObj = doc.createNestedObject("gps");
  gpsObj["valid"] = gps.location.isValid();
  gpsObj["lat"] = String(gps.location.lat(), 6);
  gpsObj["lng"] = String(gps.location.lng(), 6);
  gpsObj["satellites"] = gps.satellites.value();

  String payload;
  serializeJson(doc, payload);
  return payload;
}
```

说明：

- 该代码将设备编号、时间戳、传感器数据和 GPS 数据拼装为统一 JSON。
- 其中 `device_id` 用于后端识别设备，`timestamp` 用于时间排序，`sensors` 与 `gps` 两个对象分别承载环境数据和定位数据。
- 当前仓库中的 MQTT 主题默认为 `esp32/data`，后端也按该主题消费消息。

论文可写描述：

“设备端采用 JSON 作为上报格式，将传感器数据、定位数据和设备元信息封装为一条 MQTT 消息，保证了硬件端与后端之间的数据结构统一和接口简单化。”

### 3.4 主循环代码

来源：

- `Arduino/esp32.ino`

论文展示版代码：

```cpp
void loop() {
  readGpsStream();
  ensureWifiConnected();
  ensureMqttConnected();
  mqttClient.loop();

  unsigned long now = millis();
  if (now - lastReadAt >= SENSOR_READ_INTERVAL) {
    lastReadAt = now;

    float temperature = 0;
    float humidity = 0;
    float pressure = 0;

    if (readSensors(temperature, humidity, pressure)) {
      String payload = buildPayload("device-001", temperature, humidity, pressure);
      mqttClient.publish("esp32/data", payload.c_str());
    }
  }
}
```

说明：

- 主循环采用非阻塞方式工作，不会因为一次网络重连或一次传感器读取失败而卡死整个固件。
- 设备每隔 2 秒执行一次采样与上报，适合毕业设计演示和冷链运输短周期监控。
- `ensureWifiConnected()` 与 `ensureMqttConnected()` 负责掉线后的自动恢复。

论文可写描述：

“主循环采用定时轮询与非阻塞重连结合的方式，持续完成 GPS 读取、网络维护和数据上报，保证系统在网络抖动场景下仍能稳定运行。”

### 4.3 仅允许运输中运单接收数据的代码

来源：

- `backend/app/services/mqtt_service.py`

原始代码摘录：

```python
def _process_business(self, parsed: dict) -> dict | None:
    with SessionLocal() as db:
        device = db.scalar(
            select(Device).where(Device.device_id == parsed["device_id"]).limit(1)
        )
        if device is None:
            return None

        device.last_seen = parsed["ts"]
        db.add(device)

        active_order = db.scalar(
            select(Order)
            .where(
                Order.device_id == parsed["device_id"],
                Order.status == OrderStatus.IN_TRANSIT,
            )
            .order_by(Order.created_at.desc())
            .limit(1)
        )
        db.commit()

        if active_order is None:
            return None

        return {"order_id": active_order.order_id, "ts": parsed["ts"]}
```

说明：

- 这段代码保证只有运输中的运单才会接收设备数据。
- 如果设备未绑定运单，或者运单已经完成，则消息不会落入当前运单的时序表。
- 这个约束避免了一个设备的数据被误写入多个历史运单。

论文可写描述：

“系统通过查询设备对应的运输中运单，实现了设备数据与业务状态的绑定，避免非运输阶段的数据进入正式监控链路，从而提高了业务数据的一致性。”

```python
def insert_sensor_data(
    self,
    device_id: str,
    order_id: str,
    ts: datetime,
    temperature: float | None,
    humidity: float | None,
    pressure: float | None,
    gps_lat: float | None,
    gps_lng: float | None,
    uptime: int | None,
) -> TdengineResult:
    ok, subtable = self.ensure_subtable(device_id, order_id)
    if not ok:
        return TdengineResult(ok=False, payload={"error": "创建子表失败", "subtable": subtable})

    def to_sql_value(value):
        return "NULL" if value is None else str(value)

    ts_text = ts.strftime("%Y-%m-%d %H:%M:%S")
    sql = (
        f"INSERT INTO {self.settings.tdengine_db}.{subtable} "
        f"(ts, temperature, humidity, pressure, gps_lat, gps_lng, uptime) VALUES "
        f"('{ts_text}', {to_sql_value(temperature)}, {to_sql_value(humidity)}, "
        f"{to_sql_value(pressure)}, {to_sql_value(gps_lat)}, {to_sql_value(gps_lng)}, "
        f"{to_sql_value(uptime)})"
    )
    return self.execute_sql(sql)
```

说明：

- 该代码展示了后端如何将监控数据写入 TDengine。
- 系统会先确保运单子表存在，再执行写入语句。
- 写入字段包括时间、温度、湿度、气压、经纬度和设备运行时长，满足冷链监控的核心需求。

论文可写描述：

“系统采用 TDengine 存储高频时序监控数据，按运单建立子表，实现运输维度的数据隔离与高效查询，适合监控曲线、轨迹回放和哈希计算等后续场景。”

```python
def process_sensor_data(
    self,
    order_id: str,
    device_id: str,
    ts: datetime,
    metrics: dict[str, float | None],
) -> None:
    pending_notifications: list[dict] = []
    pending_chain_events: list[tuple[str, int]] = []
    with SessionLocal() as db:
        rules = db.scalars(
            select(AlertRule).where(AlertRule.order_id == order_id)
        ).all()
        if not rules:
            return

        for rule in rules:
            metric_name = self._enum_value(rule.metric)
            value = metrics.get(metric_name)
            if value is None:
                continue
            self._process_metric_rule(
                db=db,
                order_id=order_id,
                device_id=device_id,
                ts=ts,
                rule=rule,
                value=value,
                pending_notifications=pending_notifications,
                pending_chain_events=pending_chain_events,
            )

        db.commit()
```

说明：

- 每条监控数据进入系统后，都会按运单配置的告警规则逐项判断。
- 当前系统支持温度、湿度和气压三个维度的异常检测。
- 异常一旦触发，会进一步驱动通知推送和链上事件记录。

论文可写描述：

“后端采用基于规则的异常检测策略，将实时监控值与运单阈值配置进行逐项比较，在发现越界时同步生成异常记录、通知任务和链上事件，实现链下监测与链上存证联动。”

### 4.6 运单哈希流式计算代码

来源：

- `backend/app/services/hash_service.py`

原始代码摘录：

```python
def compute_order_hash_streaming(
    self,
    device_id: str,
    order_id: str,
    batch_size: int = 5000,
) -> str:
    hasher = hashlib.sha256()
    hasher.update(b"[")
    first = True
    cursor_ts: datetime | None = None
    cursor_offset = 0

    while True:
        result = tdengine_service.query_sensor_after_ts(
            device_id=device_id,
            order_id=order_id,
            cursor_ts=cursor_ts,
            offset=cursor_offset,
            limit=batch_size,
        )
        if not result.ok:
            if tdengine_service.is_table_not_exists(result.payload):
                break
            raise RuntimeError(f"TDengine 查询失败: {result.payload}")

        rows = tdengine_service.payload_to_rows(result.payload)
        if not rows:
            break

        for row in rows:
            normalized = self.normalize_record(row)
            chunk = json.dumps(
                normalized, ensure_ascii=False, sort_keys=True, separators=(",", ":")
            )
            if not first:
                hasher.update(b",")
            hasher.update(chunk.encode("utf-8"))
            first = False
```

说明：

- 该代码不是一次性把全部数据读入内存，而是分批流式计算哈希。
- 这样做更适合处理长时间运输产生的大规模时序数据。
- 最终哈希值会在运单完成时写入链上，用于后续数据防篡改校验。

论文可写描述：

“为避免大规模监控数据在哈希计算时占用过多内存，系统采用流式读取与增量哈希的方式，对运单全程数据进行规范化后计算 SHA-256 摘要，从而兼顾了正确性与可扩展性。”

### 4.7 链上哈希验证接口代码

```python
@router.get("/order/{order_id}/verify")
def verify_order_hash(order_id: str, ...):
    order = db.scalar(select(Order).where(Order.order_id == order_id).limit(1))
    if order is None:
        raise HTTPException(status_code=404, detail="运单不存在")
    if order.status != OrderStatus.COMPLETED:
        raise HTTPException(status_code=400, detail="运单未完成，暂无可校验哈希")

    chain_data = chain_service.get_order_hash(order_id)
    current_local_hash = _compute_current_order_hash(order)
    stored_hash = order.data_hash
    chain_hash = chain_data.get("data_hash") if chain_data else None

    return success_response(
        data={
            "local_hash": current_local_hash,
            "stored_hash": stored_hash,
            "chain_hash": chain_hash,
            "match": _normalize_hash_text(current_local_hash) == _normalize_hash_text(chain_hash),
            "local_hash_changed": _normalize_hash_text(current_local_hash)
            != _normalize_hash_text(stored_hash),
        }
    )
```

说明：

- 校验时，系统会实时从 TDengine 重新计算本地哈希，而不是只读取数据库中保存的旧快照。
- 这意味着只要时序数据库被篡改，校验接口就能第一时间发现不一致。
- `local_hash_changed` 字段进一步反映当前本地数据是否已经偏离运单完结时的快照。

论文可写描述：

“链上校验接口在校验时重新对 TDengine 中的监控数据进行哈希计算，并与链上存证值比较，因此能够有效识别时序数据库在运单完成后的追加、覆盖或篡改行为。”

### 4.8 自动重试与哈希巡检代码

来源：

原始代码摘录：

```python
async def start(self) -> None:
    await asyncio.to_thread(self._retry_failed_chain_records_once)
    await asyncio.to_thread(self._audit_order_hashes_once)
    if self._retry_task is None or self._retry_task.done():
        self._retry_task = asyncio.create_task(self._retry_loop())
    if self._hash_audit_task is None or self._hash_audit_task.done():
        self._hash_audit_task = asyncio.create_task(self._hash_audit_loop())

async def _retry_loop(self) -> None:
    while True:
        interval = self._retry_interval_seconds()
        await asyncio.sleep(interval)
        await asyncio.to_thread(self._retry_failed_chain_records_once)

async def _hash_audit_loop(self) -> None:
    while True:
        interval = self._hash_audit_interval_seconds()
        await asyncio.sleep(interval)
        await asyncio.to_thread(self._audit_order_hashes_once)
```

说明：

- 该模块承担两个后台任务：
  - 对失败的链上记录自动重试
  - 对已完成运单进行定时哈希巡检
- 自动重试用于对抗链路抖动或 RPC 短时异常。
- 哈希巡检用于在系统运行后持续监控数据是否被篡改。

论文可写描述：

“系统设计了后台完整性守护模块，一方面对失败的链上写入进行自动重试，另一方面定期对已完成运单执行链下哈希重算与链上比对，以提高链路可靠性和数据可信性。”
