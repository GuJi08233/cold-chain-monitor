# 硬件部分说明

## 1. 文档目标

本文档说明本项目硬件采集端的组成、职责、数据结构、通信方式和接入约束，便于后续进行实物接线、设备扩展、模拟调试和毕业设计答辩说明。

当前仓库中的硬件示例代码位于：

- `Arduino/esp32.ino`

该文件是一个脱敏后的 ESP32 示例固件，保留了真实项目的核心采集逻辑和上报格式，但去掉了真实 WiFi、MQTT、设备标识和生产部署凭据。

## 2. 硬件总体职责

硬件端在本系统中承担以下职责：

1. 采集冷链运输过程中的环境数据。
2. 采集车辆当前位置和卫星状态。
3. 通过 WiFi 接入网络。
4. 通过 MQTT 周期性向平台上报传感器数据。
5. 为后端的异常检测、轨迹展示、哈希计算和链上存证提供原始数据来源。

从系统角色看，硬件端不是业务管理中心，而是“数据生产端”。

## 3. 硬件清单

当前项目对应的主要硬件清单如下：

- ESP32-DevKitC
- GT-U8
- GY-SHT41
- BMP280

其中可对应理解为：

- `ESP32-DevKitC`：主控开发板
- `GT-U8`：GPS/北斗定位模块
- `GY-SHT41`：温湿度采集模块
- `BMP280`：气压采集模块

## 4. 当前示例固件的硬件组成

根据 `Arduino/esp32.ino`，当前示例固件围绕以下器件编写：

### 4.1 主控

- ESP32

主要用途：

- 运行采集固件
- 管理 WiFi 与 MQTT 连接
- 串口接收 GPS 数据
- I2C 读取温湿度和气压传感器
- 拼装 JSON 数据并上报

### 4.2 温湿度传感器

- Adafruit SHT4x

采集字段：

- `temperature`
- `humidity`

代码中会校验：

- 温度范围 `-40 ~ 85`
- 湿度范围 `0 ~ 100`

如果数据超出范围或读取失败，会把本轮温湿度视为无效。

### 4.3 气压传感器

- BMP280

默认地址：

- `0x76`

采集字段：

- `pressure`
- `altitude`

其中：

- `pressure` 会进入平台主业务链路
- `altitude` 目前更多是调试辅助字段，后端主存储模型中并未保存该值

### 4.4 GPS 模块

库依赖：

- `TinyGPS++`

串口配置：

- `GPS_RX_PIN = 16`
- `GPS_TX_PIN = 17`
- `HardwareSerial(2)`

采集字段：

- `lat`
- `lng`
- `altitude`
- `satellites`
- `valid`

后端当前真正消费的是：

- `gps.lat`
- `gps.lng`
- `gps.valid`

### 4.5 网络与时钟

固件还依赖：

- WiFi 网络
- MQTT Broker
- NTP 时间服务器

NTP 配置中默认使用：

- `ntp.aliyun.com`
- `pool.ntp.org`

默认时区为：

- `UTC+8`

这与当前项目后端默认 `APP_TIMEZONE=Asia/Shanghai` 是一致的。

## 5. 当前固件的设备模拟方式

当前仓库里的示例固件不是单设备版，而是“多设备模拟版”。

代码中固定了：

- `DEVICE_COUNT = 3`

设备 ID 列表：

- `device-001`
- `device-002`
- `device-003`

这意味着：

1. 一块 ESP32 会把同一时刻采集到的传感器数据复制为 3 份。
2. 每份数据使用不同 `device_id` 上报。
3. 适合演示多车并发接入，但不等价于真实三套硬件独立采样。

如果用于真实部署，通常应该改成：

- 一块设备对应一个真实 `device_id`

## 6. 采集与发布周期

代码中的关键周期如下：

- `STATUS_INTERVAL = 2000`
- `SENSOR_READ_INTERVAL = 2000`

即：

- 每 2 秒读取一次传感器
- 每 2 秒发布一次状态

主循环中还会进行：

- WiFi 状态检测
- MQTT 状态检测
- GPS 串口读取

## 7. 启动流程

固件启动后，主流程如下：

1. 初始化串口日志。
2. 初始化 I2C 总线。
3. 初始化 SHT4x。
4. 初始化 BMP280。
5. 初始化 GPS 串口。
6. 配置 MQTT 客户端地址和缓冲区。
7. 连接 WiFi。
8. WiFi 成功后同步 NTP 时间。
9. WiFi 成功后立即尝试连接 MQTT。
10. 进入主循环，执行连接维护、采集、上报。

## 8. 网络连接策略

### 8.1 WiFi

固件具备自动重连逻辑：

- `WIFI_RECONNECT_INTERVAL = 5000`

特点：

- 非阻塞重连
- 记录连接状态变化
- 重连成功后重新同步 NTP

### 8.2 MQTT

固件具备自动重连逻辑：

- `MQTT_RECONNECT_INTERVAL = 5000`
- `MQTT_MAX_RETRIES = 5`

策略特点：

1. 仅在 WiFi 连通时尝试 MQTT 重连。
2. 达到最大重试次数后暂停一段时间。
3. 30 秒后重置重试计数，再继续尝试。

这套机制适合运输现场网络抖动，不会因为短暂掉线直接导致程序卡死。

## 9. 时间同步策略

固件使用 NTP 来保证上报时间可信。

核心行为：

1. WiFi 首次连接成功后同步时间。
2. WiFi 断线重连成功后再次同步时间。
3. 通过 `getLocalTime()` 生成上报时间字符串。

时间字符串格式为：

- `YYYY-MM-DD HH:MM:SS`

示例：

```text
2026-03-25 14:32:10
```

若 NTP 未同步成功，固件会标记：

- `ntp_synced = false`

并把时间显示为：

- `未同步`

## 10. MQTT 上报主题

当前示例固件默认发布到：

```text
esp32/data
```

这与后端默认环境变量一致：

```env
MQTT_TOPIC=esp32/data
```

## 12. 后端真正消费的字段

后端 `mqtt_service.py` 解析后，当前主业务只消费以下字段：

- `device_id`
- `timestamp`
- `uptime`
- `sensors.temperature`
- `sensors.humidity`
- `sensors.pressure`
- `sensors.valid`
- `gps.lat`
- `gps.lng`
- `gps.valid`

后端不会写入 TDengine 的字段包括：

- `version`
- `connection`
- `sensors.altitude`
- `gps.altitude`
- `gps.satellites`

这意味着硬件端可以继续上报这些辅助字段，但平台当前不会长期持久化它们。

## 13. 后端对硬件消息的处理方式

后端在收到 MQTT 消息后执行以下步骤：

1. 解析 JSON。
2. 校验 `device_id` 是否存在。
3. 更新设备 `last_seen`。
4. 查询该设备是否存在 `in_transit` 状态运单。
5. 若没有运输中运单，则不写监控数据。
6. 若有运输中运单，则向 TDengine 写入一条时序记录。
7. 调用异常引擎判断是否超温、超湿、超压。
8. 将实时数据通过 WebSocket 推送给前端监控页面。

这意味着：

- 设备上线不等于数据一定会进入业务链路。
- 只有“设备已注册且运单处于运输中”才会真正落库。

## 14. TDengine 中的硬件数据结构

后端在 TDengine 中为每个运单写入的核心字段如下：

- `ts`
- `temperature`
- `humidity`
- `pressure`
- `gps_lat`
- `gps_lng`
- `uptime`

超级表定义为：

```sql
CREATE STABLE sensor_data (
    ts TIMESTAMP,
    temperature FLOAT,
    humidity FLOAT,
    pressure FLOAT,
    gps_lat DOUBLE,
    gps_lng DOUBLE,
    uptime INT
) TAGS (
    device_id NCHAR(64),
    order_id NCHAR(64)
)
```

因此硬件端的字段设计需要与这个存储结构保持一致。

## 15. 硬件与业务状态的关系

硬件上报本身并不控制运单状态，运单状态由平台控制。

平台侧规则是：

1. 运单 `pending` 到计划出发时间后，后端可自动转为 `in_transit`。
2. 司机也可以手动点击“提前出发”。
3. 只有 `in_transit` 运单会接收硬件数据。
4. 运单完成后，平台不再将后续上报写入该运单时序表。

所以硬件是数据提供者，而不是流程状态机的控制者。

## 16. 当前固件适合作为哪类场景

适合：

- 本地演示
- 功能联调
- 多设备模拟接入
- MQTT 与 TDengine 链路测试
- 答辩时展示设备采集端能力

不适合直接视为生产版固件的原因：

1. 真实凭据被替换为占位值。
2. 使用单板模拟多设备。
3. 目前没有本地离线缓存。
4. 目前没有 OTA 升级能力。
5. 当前缺少硬件级防拆、防断电、防伪造方案。

## 17. 如果要做真实硬件落地，建议补充

建议补充以下能力：

### 17.1 可靠性

- 离线缓存与断点补发
- 看门狗机制
- 低电压检测
- 传感器掉线告警

### 17.2 安全性

- 独立设备密钥
- MQTT TLS
- 设备身份签名
- 固件完整性校验

### 17.3 业务一致性

- 一车一设备一 `device_id`
- 与后台设备注册表严格一致
- 上报中携带更明确的固件版本和硬件版本

### 17.4 数据质量

- 采样时间与上传时间分离
- GPS 漂移过滤
- 传感器校准参数持久化

## 18. 硬件部分总结

本项目硬件端采用 ESP32 作为采集与通信中心，结合：

- SHT4x 温湿度传感器
- BMP280 气压传感器
- GPS 模块
- WiFi + MQTT 通信
- NTP 校时

构成了冷链运输监控的数据源头。

从工程角度看，当前仓库中的 `esp32.ino` 已经足以支撑“采集、上报、接入、展示、异常检测”这一整条演示链路；从产品化角度看，还需要在安全性、容错性和设备可信度方面继续加强。
