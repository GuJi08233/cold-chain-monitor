# 系统部分说明

## 1. 文档目标

本文档从系统工程视角说明本项目的整体架构、模块职责、业务流程、数据流、权限设计、部署方式和运维重点。

如果要快速理解整个项目，建议先看本文件，再分别阅读：

- `文档介绍/硬件部分.md`
- `文档介绍/智能合约部分.md`

## 2. 系统概述

本项目是一个面向冷链运输场景的“链下业务系统 + 链上存证系统”。

系统目标不是单纯展示传感器数据，而是构建一条完整链路：

1. 司机注册与审核
2. 设备注册与绑定
3. 运单创建与流转
4. 实时采集冷链监控数据
5. 自动识别异常
6. 生成消息通知
7. 关键摘要上链
8. 事后进行哈希校验与篡改发现

## 5. 单镜像部署结构

当前仓库已经调整为单镜像方案。

部署特点：

1. 构建阶段先打包前端。
2. 再将前端产物复制到后端镜像。
3. 运行时只启动一个 FastAPI 进程。
4. FastAPI 同时提供：
   - 前端静态资源
   - REST API
   - WebSocket

根目录文件：

- `Dockerfile`
- `docker-compose.yml`

默认映射：

- 宿主机 `8080`
- 容器 `8000`

## 6. 后端启动生命周期

后端入口位于：

- `backend/app/main.py`

应用启动时会依次完成：

1. `initialize_app_state()`
2. `notification_service.start()`
3. `chain_service.start()`
4. `integrity_guard_service.start()`
5. `order_lifecycle_service.start()`
6. `anomaly_engine_service.start()`
7. `mqtt_ingestion_service.start()`

这说明当前系统不是单纯的“请求响应型 API”，而是包含多个后台任务的服务型应用。

## 7. 后端核心模块

### 7.1 鉴权与用户模块

系统支持的用户角色包括：

- `super_admin`
- `admin`
- `driver`

角色区别大致如下：

- `super_admin`：最高权限，包含系统配置、用户管理等
- `admin`：运营管理权限，不包含超管专属能力
- `driver`：仅能访问自己的运单、通知、工单、个人资料

### 7.2 设备模块

设备模块负责：

1. 创建设备
2. 绑定/解绑司机
3. 展示在线状态
4. 展示最近发现的设备

设备状态与 `last_seen` 会在 MQTT 接入时持续更新。

### 7.3 运单模块

运单是系统业务主轴。

运单包含：

- 设备
- 司机
- 货物信息
- 起点终点
- 计划出发时间
- 实际出发/结束时间
- 告警规则
- 完成时哈希

### 7.4 监控模块

监控模块依赖 TDengine 查询：

- 最近或最后一条数据
- 监控总点数
- 轨迹总点数
- 完整轨迹里程
- 原始曲线
- 聚合曲线
- GPS 轨迹

其中：

1. `/monitor/{order_id}/latest` 用于订单顶部摘要和结果概览，当前采用 `no-store`，避免浏览器使用旧缓存导致统计值不一致。
2. 曲线与轨迹接口仍然保留限流、抽样和缓存优化，避免大数据量直接压垮前端图表渲染。
3. 已完成订单详情页会切换为“结果视图”，最近模式锚定到订单完成时间，自定义默认查看订单全程。

### 7.5 异常引擎

异常引擎负责：

1. 根据运单告警规则判断温度/湿度/压力是否超限
2. 生成异常开始记录
3. 跟踪峰值
4. 连续恢复后关闭异常
5. 给司机和管理员发送通知
6. 提交异常开始/结束上链

### 7.6 通知模块

通知模块会把消息：

- 存入 SQLite
- 通过 WebSocket 推送到前端

通知类型包括但不限于：

- 运单分配
- 异常开始
- 异常恢复
- 哈希异常

### 7.7 链服务

链服务负责：

1. 创建链记录
2. 异步发送链上交易
3. 存储交易哈希和区块高度
4. 失败记录自动重试
5. 读取链上摘要与异常数据

### 7.8 哈希巡检服务

完整性巡检服务负责：

1. 定时扫描已完成运单
2. 流式重算 TDengine 当前数据哈希
3. 与链上哈希进行比对
4. 发现不一致则发送通知

此外，已“测试归档”的运单会跳过自动巡检。

## 10. 业务主流程

完整业务流可以描述为：

```text
司机注册
  -> 管理员审批
  -> 创建设备并绑定司机
  -> 创建运单和告警规则
  -> 到达计划时间自动出发 / 司机手动提前出发
  -> 设备持续上报 MQTT
  -> 后端写入 TDengine
  -> 异常引擎实时判断
  -> 通知司机/管理员
  -> 异常事件上链
  -> 运单完成
  -> 流式计算运单哈希
  -> 运单哈希上链
  -> 后续支持手动或定时哈希校验
```

## 11. 运单生命周期

### 11.1 `pending`

运单已创建但尚未开始运输。

开始方式：

- 到计划时间自动转为 `in_transit`
- 或司机手动点击“提前出发”

### 11.2 `in_transit`

运输中。

此时系统会：

- 接收该设备的传感器数据
- 写入 TDengine
- 进行异常判断
- 推送实时监控

### 11.3 `completed`

司机确认到达后变为完成。

后续触发：

1. 关闭未结束异常
2. 计算整单哈希
3. 提交哈希上链

前端此时不再按“实时监控页”展示，而是切换为“结果页”语义：

- 不再显示实时模式
- 最近模式以完成时间为结束点
- 自定义默认查看运单全程
- 文案切换为“最后数据时间 / 最后定位 / 最后轨迹时间”

### 11.4 `cancelled`

运单取消。

### 11.5 `abnormal_closed`

异常关闭。

## 12. 自动出发逻辑

自动出发后台任务每 5 秒检查一次。

逻辑包括：

1. 查询所有 `planned_start <= 当前时间` 且状态为 `pending` 的运单。
2. 按计划出发时间排序。
3. 同一设备同一时刻只允许一个运单进入 `in_transit`。

这能避免设备数据被多个活跃运单同时消费。

## 13. MQTT 接入逻辑

MQTT 接入服务主要做以下事：

1. 连接 Broker
2. 订阅 `esp32/data`
3. 解码 JSON
4. 更新设备在线时间
5. 查找设备对应的运输中运单
6. 将数据写入 TDengine
7. 触发异常引擎
8. 通过 WebSocket 推送实时监控

一个关键设计点是：

- 如果设备当前没有运输中运单，则数据不会进入业务时序库

## 14. 哈希计算与校验逻辑

### 14.1 完成时快照哈希

运单完成时，系统会流式读取 TDengine 当前全部数据并计算：

- SHA-256

结果写入：

- `orders.data_hash`

然后提交链上存证。

### 14.2 手动验证

管理员或司机点击“验证哈希”时，系统会：

1. 实时重算当前 TDengine 数据哈希
2. 查询链上摘要
3. 比较当前本地哈希、完成时快照哈希、链上哈希

这样即使有人后来直接改时序库，也能被发现。

### 14.3 自动巡检

后台还会按配置周期自动巡检。

默认环境变量：

- `HASH_AUDIT_ENABLED=true`
- `HASH_AUDIT_INTERVAL_SECONDS=300`
- `HASH_AUDIT_BATCH_SIZE=20`

## 15. 上链失败自动重试

链服务默认支持自动重试失败记录。

默认环境变量：

- `CHAIN_AUTO_RETRY_ENABLED=true`
- `CHAIN_AUTO_RETRY_INTERVAL_SECONDS=30`
- `CHAIN_AUTO_RETRY_MAX_INTERVAL_SECONDS=900`
- `CHAIN_AUTO_RETRY_BATCH_SIZE=20`

适用场景：

- RPC 波动
- 网络抖动
- 交易回执超时

## 16. 测试归档能力

对于已完成且仅用于验收/篡改演示的运单，超级管理员可以执行：

- 测试归档

归档后：

- 自动哈希巡检会跳过
- 手动验证哈希仍然可用

设计原因：

- 避免测试篡改数据持续触发自动告警

实现方式：

- 复用 `system_config`
- 不修改 `orders` 表结构

## 18. 前端权限控制

前端通过 `ProtectedRoute` 和本地登录状态做权限限制。

特点：

1. 未登录用户会被重定向到 `/login`
2. 司机不能进入管理端
3. 普通 `admin` 不能进入：
   - `/admin/users`
   - `/admin/config`
4. `super_admin` 可见完整管理菜单

## 19. 系统配置管理

当前系统配置不再只依赖环境变量，也支持在前端页面修改。

配置页分组包括：

- 系统运行配置
- ETH 配置
- TDengine 配置
- MQTT 配置

超级管理员可以直接修改：

- 应用时区
- 自动重试开关与间隔
- 自动哈希巡检开关与间隔
- MQTT / TDengine / ETH 参数

环境变量仍然保留，作为数据库中未配置时的回退值。

## 20. 时间处理策略

当前系统已经统一引入：

- `APP_TIMEZONE`

默认：

- `Asia/Shanghai`

时间相关逻辑包括：

1. 运单计划时间写入
2. 自动出发时间判断
3. 手动开始/完成时间
4. MQTT 上报时间解析
5. 前端展示时间
6. 已完成订单“最近模式”的时间锚点切换为运单完成时间
7. 已完成订单“自定义模式”默认时间回填运单全程

这样可以避免之前常见的 8 小时时区偏差问题，也避免已完成订单继续以“当前系统时间”作为监控窗口终点。

## 21. 部署方式

核心变量包括：

- `TDENGINE_*`

- `/app/data/cold_chain.db`

部署后通常需要执行：

```bash

```

## 22. 系统运行中的几个关键约束

### 22.1 同一设备不允许多个运输中运单

这是避免数据归属冲突的核心约束。

### 22.2 只有运输中运单才能持续写入时序数据

这保证了监控数据与业务状态一致。

### 22.3 运单哈希只在完成后计算和上链

这保证链上记录的是“完成快照”，而不是运输过程中的中间态。

### 22.4 已归档运单不参与自动巡检

避免测试数据持续污染运维告警。

## 23. 当前系统的优势

### 23.1 优势

1. 业务链路完整
2. 实时监控与链上存证结合
3. 高低频数据分层存储合理
4. 管理端与司机端职责清晰
5. 支持自动重试和自动巡检
6. 单镜像部署方便
7. 已完成订单与运输中订单的页面语义已经分离，更适合演示和答辩说明

### 23.2 仍需增强的点

1. SQLite 更适合演示或轻量部署，生产可换 MySQL/PostgreSQL
2. TDengine 权限隔离仍需配合运维策略
3. 设备端还缺强身份认证和离线缓存
4. 链上账户安全需要额外运维措施

## 24. 适合答辩时强调的系统亮点

建议答辩时重点强调以下几点：

1. 不是单纯监控看板，而是完整业务系统。
2. 设备数据进入平台后先写时序库，再参与异常判断与哈希计算。
3. 运单完成后哈希上链，形成“链下数据 + 链上摘要”的可信结构。
4. 支持手动哈希校验和后台自动巡检，能发现事后篡改。
5. 近期又补充了自动重试、系统运行配置页、测试归档，以及已完成订单的结果视图能力。

本项目已经形成了一个较完整的冷链运输监控闭环：

- 硬件采集
- 后台接入
- 业务管理
- 实时监控
- 异常处理
- 消息通知
- 哈希存证

从工程实现角度看，它已经不是一个孤立页面或单点功能，而是一套可以演示完整业务链路的系统原型。对于“基于以太坊的冷链运输监控系统”这一题目，当前结构是自洽且有答辩说服力的。
