# 智能合约部分说明

## 1. 文档目标

本文档说明本项目链上部分的设计目标、权限模型、核心数据结构、主要函数、事件、后端集成方式，以及为什么本项目采用“链上存证 + 链下业务”的混合方案。

当前项目最终使用的合约源码位于：

- `contracts/ColdChainMonitorV3.sol`

后端运行时使用的内置 ABI 位于：

- `backend/app/contracts/cold_chain_monitor_v3_abi.json`

## 2. 合约定位

本项目智能合约的职责不是把整套冷链业务搬到链上，而是负责“关键证据存证”。

链上主要承担三类可信锚点：

1. 运单监控数据哈希
2. 异常事件起止记录
3. 司机身份锚点

因此，这个合约属于“低 gas、偏存证型合约”，而不是一个重业务、全状态链上化的合约。

## 3. 为什么采用链上存证而不是全量上链

冷链运输监控有一个很实际的问题：

- 设备数据量大
- 上报频率高
- 需要长期保存
- 查询频次高

如果把每一条温湿压/GPS 数据都写到链上，会导致：

1. gas 成本极高
2. 查询效率低
3. 前端图表展示很困难
4. 扩展性差

所以当前项目采用：

- 链下 TDengine 存原始时序数据
- 链上只保存摘要和关键事件

这是一种典型的链下存储、链上校验方案。

## 4. 合约名称与版本

当前合约：

- `ColdChainMonitorV3`

从命名可以看出，它已经是经过多轮迭代后的版本。当前版本明显加强了以下方向：

- digest 存证路径
- 低 gas 异常上链路径
- 司机身份锚点
- 更清晰的权限管理

## 5. 权限模型

合约没有引入 OpenZeppelin `Ownable`，而是自行实现了轻量权限控制。

### 5.1 核心角色

- `owner`
- `authorizedUploaders`

### 5.2 权限含义

`owner`：

- 管理员角色
- 可以转移所有权
- 可以新增或移除授权上传地址

`authorizedUploaders`：

- 允许执行写链操作的地址集合
- 只有被授权地址才能写入运单哈希、异常记录和司机锚点

### 5.3 权限修饰器

合约定义了：

- `onlyOwner`
- `onlyAuthorized`

这意味着：

- 普通地址可以读取
- 只有授权地址可以写入

### 5.4 初始化行为

合约部署时：

1. `owner = msg.sender`
2. 部署者默认加入 `authorizedUploaders`

因此部署者一开始同时拥有：

- 所有者权限
- 上传权限

### 5.5 权限管理接口

核心管理函数：

- `transferOwnership(address newOwner)`
- `addAuthorized(address account)`
- `removeAuthorized(address account)`
- `isAuthorized(address account)`

其中：

- 转移所有权时会自动授权新 owner
- 同时取消旧 owner 的上传权限
- 不允许移除当前 owner 的授权

这套策略避免了“所有权已转移但新 owner 不能写链”的尴尬状态。

## 6. 合约存储的三大模块

### 6.1 运单哈希存证

用于记录某一运单在完成时的监控数据摘要。

数据结构：

```solidity
struct OrderHashRecord {
    uint64 timestamp;
    address uploader;
    bool exists;
    bytes32 dataHashDigest;
}
```

关联存储：

- `mapping(bytes32 => OrderHashRecord) private orderHashes;`
- `mapping(bytes32 => string) private orderHashText;`

说明：

1. 业务检索时并不直接以 `string orderId` 做映射键，而是使用 `keccak256(orderId)`。
2. 当前主存储字段是 `bytes32 dataHashDigest`。
3. 为兼容旧版本，仍保留了 `orderHashText`。

### 6.2 司机身份锚点

用于记录司机身份与资料的哈希锚点，而不是把身份证等敏感明文写到链上。

数据结构：

```solidity
struct DriverAnchor {
    bytes32 driverRefHash;
    bytes32 idCommit;
    bytes32 profileHash;
    address uploader;
    uint64 updatedAt;
    bool exists;
}
```

设计意图：

- `driverRefHash`：司机引用标识哈希
- `idCommit`：身份证相关承诺值
- `profileHash`：司机资料摘要

这样做的目的，是将“司机与运单的可信关联”上链，同时避免泄露隐私原文。

### 6.3 异常事件记录

用于记录运输过程中的异常开始和结束。

数据结构：

```solidity
struct AnomalyRecord {
    uint64 startTime;
    uint64 endTime;
    int256 triggerValue;
    int256 peakValue;
    address uploader;
    bytes32 orderKey;
    bytes32 anomalyTypeKey;
    bytes32 encryptedInfoHash;
    bool closed;
    bool exists;
    bool hasInlineEncryptedInfo;
}
```

特点：

1. 以 `anomalyId` 递增编号管理。
2. 记录异常类型、起止时间、触发值、峰值。
3. 异常详情优先保存哈希，而不是敏感明文。
4. 支持旧版内联密文，也支持轻量版只上链哈希。

## 7. 运单哈希相关接口

### 7.1 兼容旧版的文本哈希接口

- `storeOrderHash(string orderId, string dataHash)`
- `getOrderHash(string orderId)`
- `verifyOrderHash(string orderId, string dataHash)`

这一路径允许直接上传字符串形式哈希。

优点：

- 兼容旧系统
- 调试更直观

缺点：

- 存储更重
- gas 相对更高

### 7.2 当前推荐的低 gas 接口

- `storeOrderHashDigest(string orderId, bytes32 dataHashDigest)`
- `getOrderHashDigest(string orderId)`
- `verifyOrderHashDigest(string orderId, bytes32 dataHashDigest)`

这是当前项目更推荐的主路径。

优点：

1. 上链数据更短
2. gas 更低
3. 后端更容易统一处理

### 7.3 防重复写入策略

无论旧接口还是 digest 接口，都不允许同一运单重复写入：

- 如果该运单哈希已存在，会触发 `OrderHashAlreadyExists`

这符合项目当前的业务假设：

- 一个完成运单只对应一份完成快照哈希

## 8. 司机锚点相关接口

### 8.1 写入接口

- `upsertDriverAnchor(orderId, driverRefHash, idCommit, profileHash)`

这是一个“upsert”语义接口。

特点：

1. 如果锚点不存在，则创建。
2. 如果锚点存在但内容不同，则更新。
3. 如果内容完全一致，则直接返回，不重复写状态。

### 8.2 查询接口

- `getDriverAnchor(orderId)`

返回：

- 三个哈希字段
- 更新时间
- 上传者
- 是否存在

## 9. 异常相关接口

### 9.1 兼容旧版异常开始接口

- `startAnomaly(orderId, anomalyType, triggerValue, startTime, encryptedInfo)`

特点：

- 会把密文原文写入链上存储
- 同时计算 `encryptedInfoHash`

这种方式可读性更强，但更耗 gas，也更不适合长期使用。

### 9.2 当前推荐的轻量异常开始接口

- `startAnomalyLite(orderId, anomalyType, triggerValue, startTime, encryptedInfoHash)`

特点：

- 只写 `encryptedInfoHash`
- 不再把异常敏感详情原文放上链
- 更适合生产环境

### 9.3 异常开始并附带司机锚点

- `startAnomalyLiteWithAnchor(...)`

这个接口会在异常开始时一起更新司机锚点，适合把“人、单、异常”三者关系同时锚定。

### 9.4 异常关闭接口

- `closeAnomaly(uint256 anomalyId, uint256 endTime, int256 peakValue)`

关闭时会校验：

1. 异常必须存在
2. 异常不能已关闭
3. `endTime` 必须晚于 `startTime`

### 9.5 异常查询接口

- `getAnomaly(anomalyId)`
- `getAnomalyMeta(anomalyId)`
- `getAnomaliesByOrder(orderId)`

这样可以实现：

- 通过运单查关联异常 ID
- 通过异常 ID 查详细信息
- 通过元信息判断是否有内联密文

## 10. 事件设计

当前合约定义的关键事件包括：

- `OwnershipTransferred`
- `AuthorizedAdded`
- `AuthorizedRemoved`
- `OrderHashStored`
- `OrderHashDigestStored`
- `DriverAnchorUpserted`
- `AnomalyStarted`
- `AnomalyStartedLite`
- `AnomalyClosed`

这些事件有两个主要作用：

1. 方便链下服务监听和审计
2. 方便后端在交易回执中提取关键结果

其中最关键的是：

- `AnomalyStartedLite`

因为后端会从事件中解析 `chain_anomaly_id`。

## 11. 合约内部辅助设计

### 11.1 字符串键转哈希键

合约内部会把：

- `orderId`
- `anomalyType`

都先转换为 `bytes32 key`：

- `_orderKey(orderId)`
- `_anomalyTypeKey(anomalyType)`

这样做有两个好处：

1. 存储键更稳定
2. 避免频繁用长字符串做主索引

### 11.2 文本字典

为了避免链上状态只剩哈希键无法回显，合约用：

- `orderIdDict`
- `anomalyTypeDict`

做了一层文本记忆。

这是一种折中方案：

- 主索引用哈希
- 展示时仍可还原文本

### 11.3 时间压缩

链上时间统一压缩为 `uint64`：

- `_asUint64(uint256 value)`

这样既节省存储，又足够满足业务生命周期需求。

## 12. 后端如何接入该合约

后端链服务位于：

- `backend/app/services/chain_service.py`

当前后端接入策略如下：

### 12.1 合约客户端构建

后端会读取：

- `ETH_RPC_URL`
- `ETH_CONTRACT_ADDRESS`
- `ETH_PRIVATE_KEY`

然后加载：

- `backend/app/contracts/cold_chain_monitor_v3_abi.json`

构建 Web3 合约实例。

### 12.2 运单完成时的行为

当运单完成后，后端会：

1. 流式读取 TDengine 中该运单全部监控数据
2. 计算本地 SHA-256 哈希
3. 将哈希保存到本地数据库 `orders.data_hash`
4. 创建一条 `ORDER_HASH` 类型链记录
5. 异步提交到链上

### 12.3 异常触发时的行为

异常引擎在检测到超限后，会：

1. 生成本地异常记录
2. 发送通知
3. 提交 `ANOMALY_START`

异常恢复或运单结束时，会：

1. 结束本地异常
2. 提交 `ANOMALY_END`

### 12.4 失败重试

当前系统已经具备：

- 上链失败自动重试

适用于：

- 网络波动
- RPC 抖动
- 等待回执超时

## 13. 哈希校验逻辑

项目现在的哈希校验不是简单比较数据库快照，而是：

1. 实时从 TDengine 重算当前本地哈希
2. 调用链上 `getOrderHash` 或 `getOrderHashDigest`
3. 再调用验证接口比对

这意味着：

- 如果有人事后直接篡改 TDengine 数据
- 页面再点“验证哈希”时可以立即发现不一致

同时，后端还有定时哈希巡检任务，会主动发现这类篡改。

## 14. 为什么合约设计成“兼容旧接口 + 新低 gas 接口”

当前合约同时保留：

- legacy 路径
- lite 路径

原因通常有三个：

1. 兼容早期版本数据和测试流程
2. 降低迁移成本
3. 为生产环境逐步切换低 gas 路径留余地

从当前后端实现来看，系统会优先探测：

- 是否存在 `storeOrderHashDigest`
- 是否存在 `startAnomalyLite`

存在则优先走新接口。

## 15. 合约的边界与局限

当前合约并不负责：

- 用户登录鉴权
- 运单创建审批
- 设备绑定
- 时序数据存储
- 图表查询
- GPS 轨迹计算

这些仍然在后端和数据库层完成。

合约只负责：

- 可信锚定
- 最小审计证据保留

## 16. 当前方案的优点

### 16.1 优点

1. 链上成本可控
2. 关键证据可信
3. 支持实时业务又保留审计能力
4. 支持异常链路与运单链路分开追踪
5. 能兼容旧版和新版调用方式

### 16.2 风险点

1. 授权上传地址若泄露，会影响写链可信度
2. 若只依赖链上摘要，原始数据仍需依赖链下安全保存
3. 若链上网络故障，需要依赖后端自动重试机制
4. 若 TDengine 被篡改，只能“发现”而不能靠合约直接阻止

## 17. 实际部署建议

部署该合约时建议：

1. owner 使用独立管理地址。
2. 后端写链账户使用单独 uploader 地址。
3. 生产环境不要长期使用部署地址直接做业务写入。
4. 对 `ETH_PRIVATE_KEY` 做严格保密。
5. RPC 使用稳定服务商并配置限流监控。

## 18. 智能合约部分总结

`ColdChainMonitorV3` 的设计重点不是把冷链运输业务完全搬上链，而是用较低成本把“运单完成哈希、异常事件、司机锚点”固化为可审计证据。

这使得本项目形成了一个很清晰的可信架构：

- 链下负责高频数据处理和业务管理
- 链上负责关键摘要和审计锚点

对于冷链运输这种“高频监控 + 低频举证”的场景，这是一种工程上更合理的实现方式。
